MySQL 事务与并发控制深度解析

MySQL 事务与并发控制深度解析

事务是数据库最核心的特性之一，理解事务与并发控制对于构建高性能、高可用的数据库应用至关重要。本文将深入探讨 MySQL 的事务机制和并发控制策略。

一、事务基础

1.1 ACID 特性

事务必须满足 ACID 四个特性：

  • 原子性（Atomicity）：事务是不可分割的工作单位，要么全部完成，要么全部不完成
  • 一致性（Consistency）：事务必须使数据库从一个一致性状态变换到另一个一致性状态
  • 隔离性（Isolation）：多个事务并发执行时，一个事务的执行不应影响其他事务的执行
  • 持久性（Durability）：一个事务一旦提交，它对数据库中数据的改变就是永久性的

二、事务隔离级别

2.1 四种隔离级别

MySQL 支持四种隔离级别，从低到高依次为：

READ UNCOMMITTED（读未提交）

  • 可以读取未提交的数据
  • 会产生脏读、不可重复读、幻读
  • 实际应用中很少使用

READ COMMITTED（读已提交）

  • 只能读取已提交的数据
  • 避免了脏读，但可能产生不可重复读和幻读
  • Oracle 默认隔离级别

REPEATABLE READ（可重复读）

  • 保证同一事务中多次读取同一数据的结果相同
  • 避免了脏读和不可重复读，可能产生幻读
  • MySQL InnoDB 默认隔离级别
  • InnoDB 通过 MVCC + Next-Key Lock 解决了幻读问题

SERIALIZABLE（可串行化）

  • 最高隔离级别
  • 强制事务串行执行
  • 避免了所有并发问题，但性能最差

2.2 查看和设置隔离级别

2.3 隔离级别与并发问题

隔离级别 脏读 不可重复读 幻读
READ UNCOMMITTED ✓ ✓ ✓
READ COMMITTED ✗ ✓ ✓
REPEATABLE READ ✗ ✗ ✓（InnoDB已解决）
SERIALIZABLE ✗ ✗ ✗

三、并发问题详解

3.1 脏读（Dirty Read）

一个事务读取了另一个事务未提交的数据。

3.2 不可重复读（Non-Repeatable Read）

一个事务内多次读取同一数据，结果不一致（数据被其他事务修改并提交）。

3.3 幻读（Phantom Read）

一个事务内多次查询，结果集的数量不一致（其他事务新增或删除了数据）。

四、锁机制

4.1 锁的分类

按锁粒度分类

  • 全局锁：锁定整个数据库实例
  • 表级锁：锁定整张表
  • 行级锁：锁定单行记录

按锁类型分类

  • 共享锁（Shared Lock，S 锁）：允许事务读一行数据
  • 排他锁（Exclusive Lock，X 锁）：允许事务删除或更新一行数据
  • 意向锁：表级锁，用于协调行锁和表锁

4.2 行锁算法

Record Lock

  • 锁定单条索引记录
  • 总是锁定索引记录，即使表没有索引也会锁定隐藏的主键索引

Gap Lock

  • 锁定索引记录之间的间隙
  • 不包括记录本身
  • 目的是防止其他事务向间隙中插入记录

Next-Key Lock

  • Record Lock + Gap Lock
  • 锁定记录本身 + 前面的间隙
  • InnoDB 在 REPEATABLE READ 隔离级别下解决幻读问题的关键

4.4 锁等待与死锁

锁等待超时

死锁

当两个或多个事务相互等待对方释放锁时，就会产生死锁。

InnoDB 会自动检测死锁，并回滚其中一个事务。

五、MVCC（多版本并发控制）

5.1 MVCC 原理

MVCC 是 InnoDB 实现高并发的核心技术，通过保存数据的历史版本来实现非阻塞读。

核心组件

  1. 隐藏字段

    • DB_TRX_ID：最近修改该行数据的事务 ID
    • DB_ROLL_PTR：指向 undo log 中该行上一个版本的指针
    • DB_ROW_ID：行 ID（如果没有主键则使用）

  2. undo log

    • 存储数据的历史版本
    • 形成版本链

  3. Read View

    • 定义了当前事务可见的数据版本
    • 包含：m_ids（活跃事务列表）、min_trx_id（最小活跃事务 ID）、max_trx_id（下一个要分配的事务 ID）、creator_trx_id（创建该 Read View 的事务 ID）

5.2 Read View 可见性判断

对于某个版本的数据，可见性判断规则：

  1. 如果 DB_TRX_ID < min_trx_id，则该版本可见（事务已提交）
  2. 如果 DB_TRX_ID >= max_trx_id，则该版本不可见（事务在 Read View 创建后才开启）
  3. 如果 DB_TRX_ID 在 m_ids 中，则该版本不可见（事务未提交）
  4. 如果 DB_TRX_ID 不在 m_ids 中，则该版本可见（事务已提交）
  5. 如果 DB_TRX_ID == creator_trx_id，则该版本可见（自己修改的）

5.3 不同隔离级别的 MVCC 实现

READ COMMITTED

  • 每次查询都生成一个新的 Read View
  • 可以看到其他事务已提交的修改

REPEATABLE READ

  • 第一次查询时生成 Read View，后续查询复用
  • 保证多次查询结果一致

六、事务最佳实践

6.1 事务使用原则

  1. 保持事务简短：事务时间越短，锁持有时间越短
  2. 避免大事务：大事务会占用大量资源，影响并发性能
  3. 合理设置隔离级别：根据业务需求选择合适的隔离级别
  4. 避免长事务：定期检查和清理长事务

6.2 死锁预防

  1. 按固定顺序访问表和行
  2. 避免长事务
  3. 尽量使用索引访问数据（避免表锁）
  4. 合理设计索引（减少锁冲突）

6.3 监控事务

七、总结

MySQL 的事务与并发控制机制是数据库高性能、高可用的基础：

  1. ACID 特性保证了数据的一致性和可靠性
  2. 隔离级别平衡了并发性能和数据一致性
  3. 锁机制实现了不同粒度的并发控制
  4. MVCC实现了非阻塞读，大幅提升了并发性能

理解这些机制，才能设计出高性能、高可用的数据库应用。
