MySQL 索引原理与优化实战

MySQL 索引原理与优化实战

索引是数据库性能优化的核心武器。本文将深入剖析 MySQL InnoDB 的索引结构，并提供实战优化技巧。

一、索引的本质

索引是帮助 MySQL 高效获取数据的数据结构。可以理解为书的目录，通过目录可以快速定位到需要的章节，而不需要逐页翻阅。

1.1 为什么使用索引

  • 减少磁盘 I/O：索引将随机 I/O 变为顺序 I/O
  • 降低数据扫描量：从全表扫描变为索引查找
  • 提供排序功能：索引本身就是有序的

1.2 索引的代价

  • 空间代价：索引需要占用磁盘空间
  • 时间代价：插入、更新、删除时需要维护索引

二、B+ 树索引结构

MySQL InnoDB 使用 B+ 树作为索引结构。

2.1 B+ 树特点

  1. 多路平衡查找树

    • 每个节点最多有 m 个子节点
    • 非叶子节点只存储键值和指针
    • 叶子节点存储所有数据，并通过指针连接

  2. B+ 树 vs B 树

    • B+ 树非叶子节点不存储数据，可以存储更多键值
    • B+ 树叶子节点通过指针连接，支持范围查询
    • B+ 树查询性能稳定，每次都要查到叶子节点

  3. B+ 树 vs 哈希表

    • 哈希表不支持范围查询和排序
    • 哈希表不支持模糊查询
    • B+ 树更适合数据库场景

2.2 InnoDB 的 B+ 树实现

InnoDB 的 B+ 树有以下特点：

  • 页（Page）：InnoDB 磁盘管理的最小单位，默认 16KB
  • 叶子节点存储完整数据：聚簇索引的叶子节点就是数据页
  • 非叶子节点存储索引键和页指针
  • 页内有序：页内的记录按主键顺序存储
  • 页间有序：通过双向链表连接

三、聚簇索引与非聚簇索引

3.1 聚簇索引

聚簇索引的叶子节点存储了完整的行数据。

特点：

  • 一张表只能有一个聚簇索引
  • 通常是主键索引
  • 如果没有主键，会选择第一个非空唯一索引
  • 如果都没有，会生成一个隐藏的 6 字节 ROWID

优点：

  • 主键查询效率高
  • 范围查询效率高（叶子节点通过链表连接）

缺点：

  • 插入速度依赖于主键顺序（建议使用自增主键）
  • 更新主键代价大（需要移动整行数据）
  • 二级索引需要回表查询

3.2 非聚簇索引（二级索引）

非聚簇索引的叶子节点存储的是索引列 + 主键值。

查询流程：

  1. 在二级索引中查找，得到主键值
  2. 在聚簇索引中查找，得到完整数据（回表）

3.3 覆盖索引

如果一个索引包含了查询所需的所有字段，就不需要回表，称为覆盖索引。

五、索引优化策略

5.1 索引设计原则

  1. 选择性高的列优先
    • 选择性 = COUNT(DISTINCT column) / COUNT(*)
    • 选择性越接近 1 越好

  1. 最左前缀原则

    • 复合索引按定义顺序使用
    • 查询条件必须包含索引的第一列

  2. 避免在索引列上使用函数

  1. 避免隐式类型转换
  1. 避免使用 !=、<>、NOT IN

5.2 索引使用场景

适合创建索引的场景

  1. WHERE 条件列
  2. JOIN 关联列
  3. ORDER BY 排序列
  4. GROUP BY 分组列
  5. DISTINCT 去重列

不适合创建索引的场景

  1. 数据量小的表
  2. 选择性低的列（如性别、状态等）
  3. 频繁更新的列
  4. 很少使用的列

九、总结

索引优化是数据库性能优化的核心：

  1. 理解 B+ 树结构，掌握聚簇索引和二级索引的区别
  2. 遵循索引设计原则，创建合适的索引
  3. 避免索引失效场景，确保索引被正确使用
  4. 使用 EXPLAIN 分析，持续优化查询性能
  5. 定期维护索引，删除无用索引，重建碎片化索引

记住：索引不是越多越好，而是要恰到好处。
