MySQL 日志系统与崩溃恢复

MySQL 日志系统与崩溃恢复

MySQL 的日志系统是保证数据安全和高可用的基石。本文将深入解析 Redo Log、Undo Log、Binlog 的原理和作用。

一、MySQL 日志体系概览

MySQL InnoDB 有多个日志系统，各司其职：

日志类型 作用 写入时机
Redo Log 保证持久性，崩溃恢复 事务提交时
Undo Log 回滚事务，MVCC 数据修改时
Binlog 主从复制，数据备份 事务提交时
Error Log 错误诊断 实时写入
Slow Query Log 慢查询分析 查询执行后
General Log 所有查询记录 查询执行时

二、Redo Log（重做日志）

2.1 为什么需要 Redo Log

问题：数据页修改是随机 I/O，性能差

解决方案：WAL（Write-Ahead Logging）

  • 先写日志，再写数据页
  • 顺序 I/O 替代随机 I/O
  • 崩溃后通过日志恢复数据

2.2 Redo Log 结构

InnoDB 的 Redo Log 采用循环写入的方式：

  • write pos：当前写入位置
  • checkpoint：检查点位置（数据已刷盘）
  • 空闲空间：checkpoint 到 write pos 之间的空间

如果 write pos 追上 checkpoint，需要暂停写入，先刷脏页。

2.4 刷盘策略

三种策略：

值 行为 性能 安全性
0 每秒刷盘一次 最高 最低（可能丢失 1 秒数据）
1 每次提交刷盘 最低 最高（不丢数据）
2 提交写入 OS 缓存，每秒刷盘 中等 中等（OS 崩溃可能丢数据）

建议：生产环境设置为 1，保证数据安全。

2.6 Redo Log 恢复流程

崩溃恢复时：

  1. 从 checkpoint 位置开始扫描 Redo Log
  2. 重做（Redo）：将已提交但未刷盘的数据重新应用
  3. 恢复到崩溃前的状态

三、Undo Log（回滚日志）

3.1 Undo Log 的作用

  1. 事务回滚：回滚未提交的事务
  2. MVCC：提供数据的历史版本，实现一致性非阻塞读

3.2 Undo Log 存储

  • InnoDB 将 Undo Log 存储在系统表空间或独立的 Undo 表空间
  • MySQL 5.6+ 支持独立的 Undo 表空间
  • MySQL 8.0+ 默认使用独立 Undo 表空间

3.3 Undo Log 类型

  1. Insert Undo Log

    • 事务插入数据时生成
    • 事务提交后可以立即删除

  2. Update Undo Log

    • 事务更新或删除数据时生成
    • 需要保留到所有读操作完成后才能删除（MVCC）

四、Binlog（归档日志）

4.1 Binlog 与 Redo Log 的区别

特性 Redo Log Binlog
存储层 InnoDB 引擎层 MySQL Server 层
格式 物理日志（数据页修改） 逻辑日志（SQL 语句或行变更）
写入方式 循环写入 追加写入
用途 崩溃恢复 主从复制、数据备份

4.2 Binlog 格式

三种格式：

  1. STATEMENT（语句模式）

    • 记录原始 SQL 语句
    • 可能导致主从不一致（如 NOW() 函数）

  2. ROW（行模式，推荐）

    • 记录每行数据的变更
    • 数据一致性最好

  3. MIXED（混合模式）

    • 默认使用 STATEMENT，特殊情况使用 ROW

为保证一致性，MySQL 采用两阶段提交：

  2. 应用已提交但未刷盘的事务

七、总结

MySQL 的日志系统是数据安全和性能的关键：

  1. Redo Log：保证持久性，WAL 技术提升性能
  2. Undo Log：支持事务回滚和 MVCC
  3. Binlog：主从复制和数据备份
  4. 两阶段提交：保证 Redo Log 和 Binlog 的一致性
