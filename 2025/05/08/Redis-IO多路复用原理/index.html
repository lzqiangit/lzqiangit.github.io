Redis IO 多路复用原理

Redis 以高性能著称，单线程却能处理数万 QPS，核心秘密就是 IO 多路复用。本文将深入解析 Redis 的事件驱动模型。

一、为什么 Redis 使用单线程

1.1 Redis 单线程的优势

Redis 的主要工作线程是单线程的，但为什么还能这么快？

  1. 纯内存操作：数据存储在内存中，读写速度极快（纳秒级）
  2. 非阻塞 IO：使用 IO 多路复用，避免阻塞等待
  3. 避免线程切换开销：没有多线程的上下文切换成本
  4. 避免锁竞争：单线程不需要锁，简化实现

1.2 Redis 的”单线程”并不完全单线程

Redis 从 4.0 开始引入了多线程，但主要用于：

  • 异步删除：UNLINK、FLUSHDB ASYNC、FLUSHALL ASYNC
  • 异步释放内存：lazy free
  • 网络读写（Redis 6.0+）：多线程网络 IO

核心命令执行仍然是单线程。

二、IO 多路复用基础

2.1 传统阻塞 IO

问题：一个连接一个线程，并发能力受限。

2.2 非阻塞 IO

问题：CPU 空转，效率低。

2.3 IO 多路复用

核心思想：一个线程监控多个文件描述符（socket），只有就绪的才进行处理。

三、Linux IO 多路复用技术

Linux 提供了三种 IO 多路复用技术：

3.1 select

特点：

  • 支持 1024 个文件描述符（可修改）
  • 每次调用需要复制 fd_set
  • 需要遍历所有 fd 查找就绪的
  • 时间复杂度：O(n)

缺点：

  • 文件描述符数量限制
  • 性能随 fd 数量增加而下降

3.2 poll

特点：

  • 没有文件描述符数量限制
  • 使用链表存储 fd
  • 时间复杂度：O(n)

缺点：

  • 仍然需要遍历所有 fd

3.3 epoll（Redis 使用）

特点：

  • 没有文件描述符数量限制
  • 只返回就绪的 fd，不需要遍历
  • 时间复杂度：O(1)
  • 支持边缘触发（ET）和水平触发（LT）

原理：

  • 使用红黑树存储所有监控的 fd
  • 使用就绪队列存储就绪的 fd
  • 通过事件回调机制，当 fd 就绪时自动加入队列

性能优势：

  • 创建连接时：O(1) 插入红黑树
  • 等待事件时：O(1) 获取就绪队列
  • 无需遍历，性能不受 fd 数量影响

4.2 文件事件（File Event）

Redis 将 socket 上的事件分为两类：

AE_READABLE（读事件）

  • 客户端连接建立（accept）
  • 客户端发送数据（read）
  • 对端关闭连接

AE_WRITABLE（写事件）

  • 发送缓冲区有空间可写

4.3 事件处理器

Redis 为不同类型的事件注册了不同的处理器：

连接应答处理器（acceptTcpHandler）

命令请求处理器（readQueryFromClient）

命令回复处理器（sendReplyToClient）

4.4 时间事件（Time Event）

五、Redis 性能优化原理

5.1 为什么 Redis 这么快

  1. 内存操作：数据在内存中，访问速度纳秒级
  2. IO 多路复用：单线程处理数万连接，无阻塞
  3. 非阻塞 IO：不会因为某个慢客户端影响整体性能
  4. 事件驱动：高效的事件循环机制
  5. 无锁设计：避免锁竞争和上下文切换

5.2 epoll 的边缘触发 vs 水平触发

Redis 使用 epoll 的**水平触发（Level Triggered，LT）**模式：

  • 水平触发：只要 socket 可读/可写，就一直触发事件
  • 边缘触发：只在状态变化时触发一次

Redis 选择 LT 的原因：

  • 实现简单
  • 不需要一次性读完所有数据
  • 兼容性好

5.3 Redis 6.0 的多线程网络 IO

Redis 6.0 引入了多线程网络 IO，提升性能：

原理：

  • 主线程负责命令执行
  • 多个 IO 线程负责网络读写
  • 减少 CPU 消耗，提升吞吐量

注意：

  • 命令执行仍然是单线程
  • 需要权衡线程数和性能
