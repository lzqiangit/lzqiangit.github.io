Redis 高可用与集群架构

Redis 高可用与集群架构

在生产环境中，Redis 的高可用和水平扩展至关重要。本文将深入解析 Redis 的主从复制、哨兵机制和集群架构。

一、Redis 高可用方案演进

1.1 单机模式

问题：

  • 单点故障，服务不可用
  • 数据丢失风险
  • 性能瓶颈

1.2 主从复制

优点：

  • 读写分离，提升读性能
  • 数据备份，降低丢失风险

缺点：

  • Master 故障需要手动切换
  • 写操作仍然是单点

1.3 哨兵模式

优点：

  • 自动故障转移
  • 监控和通知

缺点：

  • 写操作仍然是单点
  • 无法水平扩展

1.4 Redis Cluster

优点：

  • 数据分片，水平扩展
  • 自动故障转移
  • 高可用

2.2 全量复制 vs 部分复制

全量复制（SYNC）

触发场景：

  • Slave 首次连接
  • Slave 断开时间过长，复制偏移量不在 Master 的复制积压缓冲区中

流程：

  1. Master 执行 BGSAVE 生成 RDB
  2. Master 将 RDB 发送给 Slave
  3. Master 将期间的写命令缓存
  4. Slave 加载 RDB
  5. Master 发送缓存的写命令

缺点：

  • 生成 RDB 耗时（大数据集）
  • 网络传输耗时
  • Slave 阻塞

部分复制（PSYNC）

触发场景：

  • Slave 短暂断开后重连
  • 复制偏移量在 Master 的复制积压缓冲区中

流程：

  1. Slave 发送 PSYNC {runid} {offset}
  2. Master 检查 runid 和 offset
  3. 如果 offset 在缓冲区中，发送缺失的命令
  4. Slave 执行缺失的命令

优点：

  • 避免全量复制的开销
  • 快速恢复同步

三、哨兵机制详解

3.1 哨兵的作用

Redis Sentinel 提供了：

  1. 监控：监控 Master 和 Slave 是否正常运行
  2. 通知：当监控的实例出现问题时，通知管理员
  3. 自动故障转移：当 Master 故障时，自动将 Slave 提升为 Master
  4. 配置提供者：客户端通过 Sentinel 获取 Master 地址

3.2 哨兵架构

建议：至少 3 个 Sentinel，奇数个。

3.3 哨兵配置

参数说明：

  • sentinel monitor <name> <ip> <port> <quorum>
    • name：主节点名称
    • ip/port：主节点地址
    • quorum：判断主节点客观下线所需的 Sentinel 数量
  • down-after-milliseconds：主观下线时间
  • failover-timeout：故障转移超时时间
  • parallel-syncs：同时同步的新主节点数量

四、Redis Cluster 详解

4.1 Redis Cluster 特性

  • 数据分片：数据分布在多个节点
  • 高可用：每个分片有主从复制
  • 自动故障转移：主节点故障时自动切换
  • 去中心化：无中心节点，所有节点平等

4.3 集群节点通信

Redis Cluster 使用 Gossip 协议进行节点间通信：

  • PING：检测节点是否在线
  • PONG：响应 PING
  • MEET：加入集群
  • FAIL：节点下线通知

每个节点维护：

  • 集群状态（slots、nodes）
  • 其他节点的信息

4.7 故障转移

Redis Cluster 的故障转移过程：

  1. 故障检测

    • 主节点下线（主观下线 → 客观下线）
    • 集群标记节点为 FAIL

  2. 从节点选举

    • 从节点发起选举
    • 获得多数主节点投票的从节点成为新主节点

  3. 切换

    • 新主节点接管槽位
    • 其他从节点复制新主节点
    • 集群状态更新

4.9 集群限制

  1. 不支持多键操作（跨槽）
  1. 不支持多数据库
  1. Lua 脚本限制
  • 脚本涉及的 key 必须在同一个槽位
  • 使用 hash tag 解决

五、方案对比与选择

5.1 方案对比

方案 优点 缺点 适用场景
主从复制 简单、读写分离 手动故障转移 读多写少、数据量小
哨兵 自动故障转移 写单点、无法扩展 数据量中等、要求高可用
Cluster 高可用、可扩展 复杂、有限制 大数据量、高并发

5.2 选择建议

数据量 < 10GB，QPS < 10000：

  • 主从复制 + 哨兵

数据量 > 10GB 或 QPS > 10000：

  • Redis Cluster

需要跨节点事务：

  • 主从复制 + 哨兵
  • 或使用 Redis 6.2+ 的多主模式

7.2 核心原理

  • 主从复制：异步复制，全量 + 部分
  • 哨兵：监控、通知、自动故障转移
  • Cluster：数据分片、去中心化、自动故障转移

根据业务需求选择合适的方案：

  • 数据量、QPS、可用性要求
  • 运维复杂度
  • 功能限制
