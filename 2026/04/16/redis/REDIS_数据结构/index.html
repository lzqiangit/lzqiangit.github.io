String数据类型

Redis 的 STRING 类型是最基本也最常用的数据类型，本质上是一个二进制安全的字节数组（value），可以保存文本、二进制数据，或者被当作整数来做原子自增。

SDS（Simple Dynamic String）共有五种实现结构：sdshdr5、sdshdr8、sdshdr16、sdshdr32 和 sdshdr64。Redis 会根据字符串的实际长度自动选择最合适的结构类型，从而优化内存使用效率。

与传统 C 字符串不同，SDS 通过 len 属性记录字符串长度，无需遍历即可获取长度信息，且能正确处理包含空字符的二进制数据，因此适用于更广泛的场景。

什么是跳表

跳表是对链表的优化。即使链表元素有序，在链表中做查找速度还是需要遍历，速度很慢。跳表是在链表的基础上，增加了索引，使得查找速度速度达到二分查找的复杂度。

其是一种基于多层链表的概率平衡数据结构，用来实现有序集合或有序映射，能把查找、插入、删除的期望时间复杂度降到 O(log n)。直观上可以把它想成在有序的单链表上额外建立若干”跳跃”层：最底层是完整的有序链表，上层是从底层抽样得到的子链表，越高层的节点越少。查找时从最高层开始水平扫描，遇到大于目标的节点则下降一层，直至找到或确定不存在；插入时通过随机决定新节点的层数（常用的概率 p = 1/2），在每一层调整指针；删除类似——在每层把对应节点移除。

为什么用跳表实现zset？

一般可以通过跳表，二叉搜索书，B+树等来维护一个具有先后关系的集合，使用跳表是因为：

  • 相较于二叉平衡树，没有平衡开销，范围搜索性能更好
  • 相较于红黑树，实现起来更简洁，范围搜索性能也更好
  • B+树更适合需要IO操作的场景，例如文件系统、数据库等。不适用B+树是因为Redis基于内存。跳表实现更简单，添加删除时不用处理复杂的节点分裂合并

ZipList 和 QuickList

redis 压缩列表ziplist、quicklist - 知乎

ZipList（压缩列表）

ZipList 是 Redis 为了节约内存而设计的一种特殊编码的双向链表。它是一块连续的内存空间，以数组的形式存储数据，通过特殊的编码方式来压缩存储空间。

ZipList 的特点

  • 内存紧凑：所有元素连续存储，没有指针开销，适合存储小数据
  • 可变长度编码：根据实际数据大小动态调整存储空间
  • 连锁更新问题：当某个节点长度变化时，可能触发后续节点的级联更新，导致性能下降
  • 适用场景：元素数量较少且元素值较小的 List、Hash 和 Zset

QuickList（快速列表）

QuickList 是 Redis 3.2 版本引入的新数据结构，它是 ZipList 和双向链表的结合体。QuickList 本质上是一个双向链表，但每个节点都是一个 ZipList。

QuickList 的设计思路

QuickList 结合了两种数据结构的优点：

  • 双向链表的优势：支持高效的头尾插入和删除操作
  • ZipList 的优势：内存紧凑，适合存储多个小元素

QuickList 的优势

  • 平衡性能和内存：既保持了链表的 O(1) 头尾操作性能，又通过 ZipList 节省了内存
  • 避免连锁更新：将大链表拆分成多个小的 ZipList，降低了连锁更新的概率和影响范围
  • 支持压缩：可以对中间节点进行压缩，进一步节省内存
  • 默认实现：Redis 3.2+ 版本中，List 类型默认使用 QuickList 作为底层实现
