知识集与 Skill

一、Skill与知识集

1.1 Skill

定义：Skil 是一种标准化的知识封装格式，让大模型在特定领域拥有专家级的执行能力。

本质：一个文件夹，里面装着”该怎么做”的指令、”可以用什么工具”的脚本、以及”参考什么标准”的文档。

核心特征：

  • 可执行性：包含明确的执行步骤和操作流程
  • 模块化：独立的功能单元，可以单独使用
  • 可组合性：可以与其他 Skill 组合使用
  • 聚焦性：专注于解决特定问题
  • 简洁性：不应过于庞大，保持清晰

1.2 什么是知识集？

定义：知识集是用户自定义的”知识地图”，涵盖平台资源配置（iCode 代码库、iCafe 需求空间、iAPI 文档、如流群 ID、iDataset 等）以及背景知识、工作流、团队规范、F&Q 等。

本质：配置索引与数据实体的双层结构：

  • 配置索引层（knowledge_book/）：用户定义的知识地图，记录各类资源的存储路径和访问方式
  • 数据实体层（knowledge/ 下的其他目录）：公司内部各平台的知识集合（iCode 代码库、知识库文档、iCafe 需求卡片、iAPI 接口文档等）

核心特征：

  • 存储性：存储事实数据、文档、配置信息
  • 检索性：通过关键词搜索定位特定知识
  • 组织性：按领域、类型、项目等维度组织知识
  • 时效性：知识具有创建/修改时间，需要关注时效性

1.3 对比

维度 程序性知识 陈述性知识
定义 知道怎么做 知道是什么
例子 “如何根据症状选择抗生素” “阿莫西林的半衰期是1.5小时”
特征 有判断流程、需要完整理解逻辑 纯事实数据、只需检索匹配
适合载体 Skills 知识集
对比维度 Skill 知识集
核心作用 定义”会做什么、怎么做” 存储”是什么、在哪里”
内容类型 方法论、流程、决策规则 事实数据、文档、配置
AI 使用方式 加载到上下文，理解并执行 检索匹配，返回具体内容
容量约束 受上下文窗口限制（建议几百条以内） 理论上无上限
更新方式 修改 SKILL.md、脚本、配置文件 通过知识-fetch 技能拉取最新内容
版本管理 通过 dodo-skill-publish 发布到个人技能库 知识内容随源平台更新
触发机制 基于用户意图自动匹配（Skill Matcher） 基于关键词主动检索

2.3 判断标准：该用 Skill 还是知识集？

核心判断问题：”AI 需要理解这个方法的完整逻辑，还是需要查到具体数值？”

经验内容 判断 归属
某 API 的请求参数和返回格式 查到即可 知识集
如何根据错误码定位问题根因 需要理解判断流程 Skill
公司所有客户的联系方式 纯数据检索 知识集
客户分级标准和服务策略 决策规则 Skill
某试剂的最佳配比 1:2:3 查到即可 知识集
如何根据目标产率选择配比策略 需要理解方法论 Skill
团队代码规范文档 查阅参考 知识集
如何进行代码评审的完整流程 执行步骤 Skill

1.4 规模约束参考

数量级 推荐方案
几十条方法 直接写成 Skills
几百条方法 Skills 分层 + 目录索引
几千条以上 必须用知识集

2.3 Skill 设计五原则

原则 说明 反例
边界明确 明确正向条件和负向条件 只写”代码评审”而不写何时触发
输入输出结构化 用函数签名方式定义 Input/Output “帮用户跑测试并返回结果”
步骤明确、可执行 每步都是可执行的动作 “检查 PR，运行测试，然后更新状态”
失败策略完备 定义不同失败情况的处理方式 只写成功路径
职责绝对单一 一个 Skill 只做一件事 一个 Skill 同时完成测试+通知+更新状态

3.4 Skill 编写常见误区

误区 正确理解
Skill 等同于一段 Prompt Skill 是长期复用、输入输出明确的能力模块，Prompt 是临时性交互
Skill 是写给人看的文档 Skill 目标是”下达指令”，应使用模型可解析的结构化语言
Skill 越复杂越强大 职责单一、边界清晰的 Skill 更容易被正确触发和稳定执行

4.2 知识本（Knowledge Book）配置

知识本是用户自定义的知识地图，记录：

  • 平台资源配置（iCode 代码库、iCafe 需求空间、iAPI 文档、如流群 ID）
  • 背景知识、工作流、团队规范、F&Q
  • 各类资源的存储路径和访问方式

索引文件：knowledge_book_usage.md 记录已配置知识本的清单及存储路径。

4.3 知识集使用原则

  1. 主动检索：当任务信息缺失或无法准确回答时，必须主动检索知识本
  2. 全量搜索：默认在全量知识本中搜索，不要凭主观判断跳过某个知识本
  3. 时效性校验：关注知识的创建/修改时间，内容存在不代表可用
  4. 动态拉取：若发现现有知识与用户 Query 不符或过时，调用 knowledge-fetch 技能拉取最新知识

5.2 已安装 Skill 更新最新版本

场景：从远端技能托管平台安装的 Skill 有新版本

更新步骤：

  1. 使用 skill-recommender Skill 查看可用 Skills 列表
  2. 找到目标 Skill，查看版本信息和更新日志
  3. 重新安装最新版本（覆盖旧版本）

注意：

  • 本地修改过的 Skill 重新安装会被覆盖
  • 建议在更新前备份本地修改内容

6.1 协作架构

6.2 典型场景示例

场景：企业内部问答系统

角色 内容
知识集 存储几万份产品文档、技术规范、历史工单
Skills 定义”如何根据问题类型选择检索范围”、”如何判断检索结果是否足够回答问题”、”如何处理跨文档的关联查询”

结论：Skills 让 AI 变成一个更聪明的检索者，但被检索的内容还是在知识集里。

6.3 协作最佳实践

  1. 先分类：把知识资产过一遍，区分方法论和事实数据
  2. 方法论精简：高频使用、包含不可替代判断逻辑的才放 Skills
  3. 目录索引：按领域组织目录，每个目录下放内容概要
  4. 预处理格式：知识集中的 PDF、Excel 等提前转成纯文本或 Markdown

七、最佳实践总结

7.1 Skill 最佳实践

维度 实践建议
命名规范 小写字母、数字和连字符，使用动名词，长度不超过 64 字符
描述规范 第三人称、包含核心功能与触发时机关键词、长度不超过 1024 字符
触发条件 明确正向条件和负向条件（Use this skill when / Do NOT use this skill when）
渐进式披露 SKILL.md 保持简洁（500 行以内），详细内容拆分到 references/
失败策略 明确定义不同失败情况下的处理方式
职责单一 一个 Skill 只做一件事，对应一个核心动作动词
脚本健壮性 显式处理错误、输出自解释的日志、避免魔法数字

7.2 知识集最佳实践

维度 实践建议
分类存储 按领域、类型、项目等维度组织知识
时效性管理 定期检查知识创建/修改时间，过时内容及时更新
格式统一 知识内容尽量使用 Markdown，便于检索和解析
索引完善 每个目录下放内容概要，帮助快速定位
动态拉取 发现知识过时，及时调用 knowledge-fetch 拉取最新内容

九、总结

一句话总结：Skills 是方法论，知识集是数据。方法论告诉 AI 怎么思考和检索，数据提供具体的事实支撑。

判断标准不是规模，而是知识性质：

  • 需要 AI 理解完整判断流程 → Skills
  • 只需检索匹配返回 → 知识集

最优解通常是两者结合：用 Skills 封装智能检索和决策逻辑，用知识集承载大规模事实数据。
