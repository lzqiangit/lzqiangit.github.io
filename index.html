知识集与Skill

发表于 2026-04-26 分类于 Harness

1、定义角度

当前许多关于Harness（Harness Engineering）的讨论和实践，但不同方向的从业者所说的 Harness 角度是不同的。

  • 对于优化 Agent 开发体验的开发者，其所述的 Harness，往往聚焦于模型之外，让 Agent 能感知、规划、调用工具、执行动作、接收反馈并持续完成任务的运行与控制系统。它解决的是：模型怎么访问工具和环境、怎么持久化状态跨会话继续工作、怎么控制上下文膨胀、怎么做规划执行验证和重试、多个 Agent 怎么协作分工。如果借用 LangChain 的表达，可以写成 Agent = Model + Harness。
  • 对于利用各种 Agent 提升开发效率，并尝试构建 AI 时代的软件工程最佳实践的开发者，Harness 则是围绕 Agent 重构代码库、文档、测试、可观测性和开发流程，使 Agent 能在真实软件工程里稳定协作、验证和交付的外层系统。其需要解决的问题是诸如：repo 是否对 Agent 可读、docs 和规则是否结构化、tests / CI / review / 验收是否能进入 agent loop、UI 和日志指标 trace 是否对 Agent 可见、团队流程是否支持 agent-first 的开发方式。

总结就是Agent Harness让 Agent 跑起来；面向 Agent 软件工程的 Harness 让 Agent 在真实工程里产生价值。两者最终拥有一个共同目标：AI-first。

Most companies bolt AI onto their existing process … Efficiency goes up 10 to 20 percent. Nothing structurally changes. That is AI-assisted.

AI-first means you redesign your process, your architecture, and your organization around the assumption that AI is the primary builder.

2、目标挑战

Harness 的最终目标是实现人类掌舵，Agent 执行。在传统团队里，很多价值主要来自工程师亲手完成实现：自己查代码、自己跑环境、自己改逻辑、自己验证结果、自己兜底隐性知识。在面向 Agent 的工程环境里，人的高价值工作会逐渐迁移到：定义目标、拆解任务、提供清晰参考、设计工程环境、设定边界和约束、判断风险、处理异常、沉淀规则。与此同时，Agent 的价值也不在”替代工程师”，而在于承担越来越多可定义、可执行、可验证、可迭代的执行工作。

当 Agent 真正进入研发流程之后，开发者面对的不再只是”它会不会写代码”，而是：

  • 它能看到什么
  • 它能理解什么
  • 它能做什么
  • 它怎么知道自己做得对不对
  • 系统怎么防止它长期失控
  • 人在这个过程中应该扮演什么角色

换句话说，真正发生变化的不是”多了一个工具”，而是——我们的研发系统，开始需要对 Agent 友好。
这也是 Harness Engineering 真正重要的地方。它不是 prompt 小技巧，也不是某个工具的使用心得，而是在讨论：当研发系统开始围绕 Agent 重构时，我们到底应该建设什么。

3、Harness 的工作方式和四层能力

Harness Agent 的工作方式可以总结为 Perceive / Decide / Act / Feedback 四个环节。

  • Perceive（先看见世界）：Agent 需要先读取任务、代码、文档、历史决策、环境状态和已有结果。它的一切判断都依赖当前能看到什么。所以第一层问题不是”模型强不强”，而是系统有没有把关键上下文以 Agent 能消费的方式暴露出来。
  • Decide（判断下一步做什么）：读完上下文之后，Agent 要判断先查什么、先改什么、先验证什么、什么时候该停、什么时候应该升级给人。在复杂任务里，它还要涉及任务拆解和推进顺序安排。
  • Act（真的采取动作）：Agent 不只是输出建议，还要改代码、跑脚本、执行测试、调用工具、查询日志和指标、触发工作流、发起 PR、响应评论。只有能真正进入动作层，它才不是一个”会说怎么做”的模型，而是一个能参与工程闭环的执行体。
  • Feedback（根据结果继续修正）：动作发生之后，Agent 还要知道测试有没有通过、脚本有没有报错、页面是不是正常、性能有没有退化、指标有没有变化、改动到底有没有生效。这些结果如果不能回流，下一轮决策就无从谈起。

可以理解为 “感知 → 决策 → 行动 → 反馈 → 再感知” 的闭环系统。将闭环翻译成工程语言，我们就得到了其真正要建设的是四层能力：

Agent 运行闭环 对应工程能力
Perceive 让 Agent 读懂
Decide + Act 让 Agent 可执行
Feedback 让 Agent 有反馈闭环
长期稳定运行 让 Agent 可治理

知识集与 Skill

一、Skill与知识集

1.1 Skill

定义：Skil 是一种标准化的知识封装格式，让大模型在特定领域拥有专家级的执行能力。

本质：一个文件夹，里面装着”该怎么做”的指令、”可以用什么工具”的脚本、以及”参考什么标准”的文档。

核心特征：

  • 可执行性：包含明确的执行步骤和操作流程
  • 模块化：独立的功能单元，可以单独使用
  • 可组合性：可以与其他 Skill 组合使用
  • 聚焦性：专注于解决特定问题
  • 简洁性：不应过于庞大，保持清晰

1.2 什么是知识集？

定义：知识集是用户自定义的”知识地图”，涵盖平台资源配置（iCode 代码库、iCafe 需求空间、iAPI 文档、如流群 ID、iDataset 等）以及背景知识、工作流、团队规范、F&Q 等。

本质：配置索引与数据实体的双层结构：

  • 配置索引层（knowledge_book/）：用户定义的知识地图，记录各类资源的存储路径和访问方式
  • 数据实体层（knowledge/ 下的其他目录）：公司内部各平台的知识集合（iCode 代码库、知识库文档、iCafe 需求卡片、iAPI 接口文档等）

核心特征：

  • 存储性：存储事实数据、文档、配置信息
  • 检索性：通过关键词搜索定位特定知识
  • 组织性：按领域、类型、项目等维度组织知识
  • 时效性：知识具有创建/修改时间，需要关注时效性

九、总结

一句话总结：Skills 是方法论，知识集是数据。方法论告诉 AI 怎么思考和检索，数据提供具体的事实支撑。

判断标准不是规模，而是知识性质：

  • 需要 AI 理解完整判断流程 → Skills
  • 只需检索匹配返回 → 知识集

最优解通常是两者结合：用 Skills 封装智能检索和决策逻辑，用知识集承载大规模事实数据。

String数据类型

Redis 的 STRING 类型是最基本也最常用的数据类型，本质上是一个二进制安全的字节数组（value），可以保存文本、二进制数据，或者被当作整数来做原子自增。

SDS（Simple Dynamic String）共有五种实现结构：sdshdr5、sdshdr8、sdshdr16、sdshdr32 和 sdshdr64。Redis 会根据字符串的实际长度自动选择最合适的结构类型，从而优化内存使用效率。

与传统 C 字符串不同，SDS 通过 len 属性记录字符串长度，无需遍历即可获取长度信息，且能正确处理包含空字符的二进制数据，因此适用于更广泛的场景。

什么是跳表

跳表是对链表的优化。即使链表元素有序，在链表中做查找速度还是需要遍历，速度很慢。跳表是在链表的基础上，增加了索引，使得查找速度速度达到二分查找的复杂度。

其是一种基于多层链表的概率平衡数据结构，用来实现有序集合或有序映射，能把查找、插入、删除的期望时间复杂度降到 O(log n)。直观上可以把它想成在有序的单链表上额外建立若干”跳跃”层：最底层是完整的有序链表，上层是从底层抽样得到的子链表，越高层的节点越少。查找时从最高层开始水平扫描，遇到大于目标的节点则下降一层，直至找到或确定不存在；插入时通过随机决定新节点的层数（常用的概率 p = 1/2），在每一层调整指针；删除类似——在每层把对应节点移除。

为什么用跳表实现zset？

一般可以通过跳表，二叉搜索书，B+树等来维护一个具有先后关系的集合，使用跳表是因为：

  • 相较于二叉平衡树，没有平衡开销，范围搜索性能更好
  • 相较于红黑树，实现起来更简洁，范围搜索性能也更好
  • B+树更适合需要IO操作的场景，例如文件系统、数据库等。不适用B+树是因为Redis基于内存。跳表实现更简单，添加删除时不用处理复杂的节点分裂合并

ZipList 和 QuickList

redis 压缩列表ziplist、quicklist - 知乎

ZipList（压缩列表）

ZipList 是 Redis 为了节约内存而设计的一种特殊编码的双向链表。它是一块连续的内存空间，以数组的形式存储数据，通过特殊的编码方式来压缩存储空间。

ZipList 的特点

  • 内存紧凑：所有元素连续存储，没有指针开销，适合存储小数据
  • 可变长度编码：根据实际数据大小动态调整存储空间
  • 连锁更新问题：当某个节点长度变化时，可能触发后续节点的级联更新，导致性能下降
  • 适用场景：元素数量较少且元素值较小的 List、Hash 和 Zset

QuickList（快速列表）

QuickList 是 Redis 3.2 版本引入的新数据结构，它是 ZipList 和双向链表的结合体。QuickList 本质上是一个双向链表，但每个节点都是一个 ZipList。

QuickList 的设计思路

QuickList 结合了两种数据结构的优点：

  • 双向链表的优势：支持高效的头尾插入和删除操作
  • ZipList 的优势：内存紧凑，适合存储多个小元素

QuickList 的优势

  • 平衡性能和内存：既保持了链表的 O(1) 头尾操作性能，又通过 ZipList 节省了内存
  • 避免连锁更新：将大链表拆分成多个小的 ZipList，降低了连锁更新的概率和影响范围
  • 支持压缩：可以对中间节点进行压缩，进一步节省内存
  • 默认实现：Redis 3.2+ 版本中，List 类型默认使用 QuickList 作为底层实现

    • Future

1.2, 线程池核心参数

核心线程数，最大线程数和等待队列是最核心的三个参数

1.2.1, 线程数

对于CPU密集型任务：n（ncpu核心数）(不推荐n+1是因为，对于缺页等中断，还是会需要cpu处理，计算密集型任务cpu始终是瓶颈）

对于IO密集型任务：m*n（m>1）一般推荐m为2

线程数更严谨的计算的方法应该是：最佳线程数 = N（CPU 核心数）∗（1+WT（线程等待时间）/ST（线程计算时间）），其中 WT（线程等待时间）=线程运行总时间 - ST（线程计算时间）

1.2.2，阻塞队列

  • LinkedBlockingQueue：无界（本节无界即为Integer.MAX_VALUE）
  • SynchronousQueue：容量为0，用于CachedThreadPool （无限核心线程数）
  • DelayedWorkQueue：通过堆实现的，按处理时间排序的，容量无限的队列（ScheduledThreadPool 使用）
  • ArrayBlockingQueue：可设定最大容量

1.2.3，拒绝策略

  • 直接抛出异常
  • 让调用的线程执行（如果原线程没有结束）
  • 不抛出异常
  • 丢弃最早的任务

默认的拒绝策略：ThreadPoolExecutor.AbortPolicy：抛出 RejectedExecutionException来拒绝新任务的处理。

1.2, 实现原理

面试题：任务达到核心线程数，是先扩容线程数还是先进阻塞队列

答：先添加到阻塞队列，后扩容

submit和execute的区别

答：execute不支持返回结果，不能查看运行情况。submit实际上是将任务包装成Future后提交给execute的

java可以通过ThreadPoolExecutor自定义线程池，这是推荐的做法。java也提供了一些内置的线程池，可以通过Executors的工厂方法创建

  • FixedThreadPool
  • SingleThreadExecutor
  • CachedThreadPool （无限线程数，阻塞队列长为0）
  • ScheduledThreadPool （定时任务https://javaguide.cn/system-design/schedule-task.html）

2，CompletableFuture

CompletableFuture 是对Future的增强，除了提供了更为好用和强大的 Future 特性之外，还提供了函数式编程、异步任务编排组合（可以将多个异步任务串联起来，组成一个完整的链式调用）等能力。

2.0，Future

Future 是用来表示一个异步计算的结果占位符——你可以通过它查询任务是否完成、取消任务、并在需要时阻塞等待并获取计算结果。常见实现有通过 ExecutorService.submit 返回的实现和 FutureTask；在现代异步场景下更常用 CompletableFuture 来支持链式/回调式处理。

2.1，构造

  • new
  • 工厂 supplyAsync （有返回值的任务），runAsync（无返回值的任务）（第一个参数是任务，第二个可选参数是线程池）

推荐通过自定义线程池构造

2.2, 处理异步结果

  • thenApply:传入,传入
  • thenAccept:只传入,不传出
  • thenRun: 不关心上一步结果
  • whenComplete:在完成时做“副作用”处理（能同时看到结果或异常），不会改变原结果（返回与上一步相同的值类型），常用于记录/清理/观察异常

通常在“完成前置阶段的线程”上执行

带 Async 的变体（thenApplyAsync / thenAcceptAsync / thenRunAsync / whenCompleteAsync）会在异步执行环境中运行：默认使用 ForkJoinPool.commonPool()，或者使用你传入的 Executor（thenApplyAsync(fn, executor)）运行。

  • 异常处理:handle

  • thenCombine会在两个任务都执行完成后，把两个任务的结果合并。两个任务是并行执行的
  • acceptEither 两个任务完成一个进行定义的任务
  • CompletableFuture 的 allOf()这个静态方法来并行运行多个 CompletableFuture

synchronized

实现原理： 通过JVM虚拟机的monitorenter 和 monitorexit指令，在同步代码块入口通过monitorenter获取对象监视器的持有权，退出的时候通过monitorexit释放持有权。获取持有权就是获取锁，释放持有权就是释放锁。此外，wait/notify也是依赖于对象监视器的，所以这两个方法必须在synchronized代码块中调用.

锁升级： 锁升级机制的核心目标是在保证线程安全的前提下，根据实际的线程竞争情况，智能地选择最合适的锁实现。，synchronized无论有无竞争，直接使用重量级锁。这种锁依赖于操作系统的互斥量（Mutex Lock），导致每次加锁和解锁都需要进行用户态到内核态的切换，开销很大。

偏向锁（CAS，适合单个线程重复获取锁）

– 出现竞争–>

轻量级锁（自旋CAS，适合持有锁时间较短，两个线程竞争，没有用户态内核态切换）

– 自旋次数过多/竞争线程过多–>

重量级锁（操作系统的Mutex，需要操作系统切换）

锁监视器Monitor，当升级为重量级锁的时候，对象头的Mark word将指向Monitor对象

可重入锁： synchronized是可重入锁，其通过 对象监视器 的”持有对象” 和 “计数器”属性字段实现
