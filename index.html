Skills：AI Agent 的能力扩展机制

一、背景

1.1 AI Agent 的发展演进

随着大语言模型（LLM）技术的快速发展，AI 系统正从单纯的对话助手向**自主智能体（Agent）**演进。传统的 LLM 虽然具备强大的语言理解和生成能力，但在面对复杂任务时仍存在以下局限：

  • 工具调用受限：无法直接访问外部系统、API 或执行具体操作
  • 领域知识固化：训练数据截止后无法获取最新信息
  • 任务分解困难：难以将复杂目标拆解为可执行的子任务
  • 上下文管理薄弱：长周期任务中容易丢失关键状态信息

为解决这些问题，Skills（技能） 作为一种结构化的能力封装机制应运而生。

1.2 Skills 的诞生

Skills 概念最早由 Anthropic 在 Claude Agent 系统中提出，随后被 OpenAI、Google DeepMind 等机构广泛采纳。其核心思想是：将特定领域的能力封装为可复用、可组合的标准化模块，使 AI Agent 能够像人类专家一样”掌握”多种专业技能。

二、作用

2.1 核心功能

2.1.1 能力扩展

Skills 为 AI Agent 提供了以下扩展能力：

  • 工具集成：封装 API 调用、数据库查询、文件操作等技术能力
  • 领域专业化：注入特定行业知识（如医疗诊断、法律分析、代码审查）
  • 流程自动化：预定义复杂任务的执行步骤和决策逻辑
  • 记忆增强：维护长期状态、用户偏好和历史交互记录

2.1.2 模块化设计

每个 Skill 独立封装，支持：

  • 按需加载：根据任务类型动态激活相关技能
  • 热插拔：运行时添加或删除技能模块
  • 版本管理：独立更新单个技能而不影响其他模块

2.2 实际应用场景

场景 所需 Skills 效果提升
软件开发 代码生成、单元测试、Bug 修复 开发效率提升 3-5 倍
数据分析 SQL 查询、可视化、统计建模 分析准确率提高 40%
内容创作 SEO 优化、多语言翻译、风格适配 产出质量更稳定
客户服务 知识库检索、工单分类、情绪识别 响应速度提升 60%

四、最佳实践

4.1 设计原则

五、总结

Skills 作为 AI Agent 的核心扩展机制，正在重塑人机协作的方式。通过模块化、标准化的能力封装，我们能够实现：

  1. 更高的灵活性：按需组合技能应对不同场景
  2. 更好的可维护性：独立开发和测试每个技能
  3. 更强的可扩展性：轻松集成新工具和领域知识
  4. 更低的成本：复用已有技能，避免重复开发

未来发展方向包括：

  • 自动技能发现：Agent 自主学习新技能
  • 技能市场：社区共享和交易 Skills
  • 自适应优化：根据使用反馈自动调优
  • 跨平台兼容：统一的 Skill 标准协议

掌握 Skills 的设计和应用，将是构建下一代智能应用的关键能力。

Java线程同步

发表于 2026-04-16 分类于 Java ， JUC

synchronized

实现原理： 通过JVM虚拟机的monitorenter 和 monitorexit指令，在同步代码块入口通过monitorenter获取对象监视器的持有权，退出的时候通过monitorexit释放持有权。获取持有权就是获取锁，释放持有权就是释放锁。此外，wait/notify也是依赖于对象监视器的，所以这两个方法必须在synchronized代码块中调用.

锁升级： 锁升级机制的核心目标是在保证线程安全的前提下，根据实际的线程竞争情况，智能地选择最合适的锁实现。，synchronized无论有无竞争，直接使用重量级锁。这种锁依赖于操作系统的互斥量（Mutex Lock），导致每次加锁和解锁都需要进行用户态到内核态的切换，开销很大。

偏向锁（CAS，适合单个线程重复获取锁）

– 出现竞争–>

轻量级锁（自旋CAS，适合持有锁时间较短，两个线程竞争，没有用户态内核态切换）

– 自旋次数过多/竞争线程过多–>

重量级锁（操作系统的Mutex，需要操作系统切换）

锁监视器Monitor，当升级为重量级锁的时候，对象头的Mark word将指向Monitor对象

可重入锁： synchronized是可重入锁，其通过 对象监视器 的”持有对象” 和 “计数器”属性字段实现

Java线程池和任务编排

发表于 2026-04-16 分类于 Java ， JUC

1，线程池

1.1, Executor 框架

  • 任务
    • Callable（有返回值)
    • runnable（无返回值）

Runnable 接口不会返回结果或抛出检查异常，但是 Callable 接口可以

  • 执行任务
    • Executor（接口）
  • 异步执行结果
    • Future
    • CompletableFuture

1.2, 线程池核心参数

核心线程数，最大线程数和等待队列是最核心的三个参数

1.2.1, 线程数

对于CPU密集型任务：n（ncpu核心数）(不推荐n+1是因为，对于缺页等中断，还是会需要cpu处理，计算密集型任务cpu始终是瓶颈）

对于IO密集型任务：m*n（m>1）一般推荐m为2

线程数更严谨的计算的方法应该是：最佳线程数 = N（CPU 核心数）∗（1+WT（线程等待时间）/ST（线程计算时间）），其中 WT（线程等待时间）=线程运行总时间 - ST（线程计算时间）

1.2.2，阻塞队列

  • LinkedBlockingQueue：无界（本节无界即为Integer.MAX_VALUE）
  • SynchronousQueue：容量为0，用于CachedThreadPool （无限核心线程数）
  • DelayedWorkQueue：通过堆实现的，按处理时间排序的，容量无限的队列（ScheduledThreadPool 使用）
  • ArrayBlockingQueue：可设定最大容量

1.2.3，拒绝策略

  • 直接抛出异常
  • 让调用的线程执行（如果原线程没有结束）
  • 不抛出异常
  • 丢弃最早的任务

默认的拒绝策略：ThreadPoolExecutor.AbortPolicy：抛出 RejectedExecutionException来拒绝新任务的处理。

1.2, 实现原理

面试题：任务达到核心线程数，是先扩容线程数还是先进阻塞队列

答：先添加到阻塞队列，后扩容

submit和execute的区别

答：execute不支持返回结果，不能查看运行情况。submit实际上是将任务包装成Future后提交给execute的

1.3，内置线程池

java可以通过ThreadPoolExecutor自定义线程池，这是推荐的做法。java也提供了一些内置的线程池，可以通过Executors的工厂方法创建

  • FixedThreadPool
  • SingleThreadExecutor
  • CachedThreadPool （无限线程数，阻塞队列长为0）
  • ScheduledThreadPool （定时任务https://javaguide.cn/system-design/schedule-task.html）

2，CompletableFuture

CompletableFuture 是对Future的增强，除了提供了更为好用和强大的 Future 特性之外，还提供了函数式编程、异步任务编排组合（可以将多个异步任务串联起来，组成一个完整的链式调用）等能力。

2.0，Future

Future 是用来表示一个异步计算的结果占位符——你可以通过它查询任务是否完成、取消任务、并在需要时阻塞等待并获取计算结果。常见实现有通过 ExecutorService.submit 返回的实现和 FutureTask；在现代异步场景下更常用 CompletableFuture 来支持链式/回调式处理。

2.1，构造

  • new
  • 工厂 supplyAsync （有返回值的任务），runAsync（无返回值的任务）（第一个参数是任务，第二个可选参数是线程池）

推荐通过自定义线程池构造

2.2, 处理异步结果

  • thenApply:传入,传入
  • thenAccept:只传入,不传出
  • thenRun: 不关心上一步结果
  • whenComplete:在完成时做“副作用”处理（能同时看到结果或异常），不会改变原结果（返回与上一步相同的值类型），常用于记录/清理/观察异常

通常在“完成前置阶段的线程”上执行

带 Async 的变体（thenApplyAsync / thenAcceptAsync / thenRunAsync / whenCompleteAsync）会在异步执行环境中运行：默认使用 ForkJoinPool.commonPool()，或者使用你传入的 Executor（thenApplyAsync(fn, executor)）运行。

  • 异常处理:handle

2.3, 组合排序

  • thenCompose按顺序链接两个 CompletableFuture 对象，实现异步的任务链
  • thenCombine会在两个任务都执行完成后，把两个任务的结果合并。两个任务是并行执行的
  • acceptEither 两个任务完成一个进行定义的任务
  • CompletableFuture 的 allOf()这个静态方法来并行运行多个 CompletableFuture
